# Implementation Plan: PM Kit — Product Management Commands & Templates

**Branch**: `001-pm-kit-commands` | **Date**: 2026-03-28 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/001-pm-kit-commands/spec.md`

## Summary

pm-kit is a set of four Claude Code commands and four Markdown templates that bring Amazon's Working
Backwards (PR/FAQ) methodology to spec-driven development projects. It complements spec-kit by managing
the product narrative (domain context, press releases, FAQs, and a release-independent requirements
overview) that precedes and informs engineering specifications.

The implementation is entirely Markdown — commands are `.claude/commands/pm-*.md` files, templates
are `.product/templates/*.md` files. No runtime code or external dependencies. Installation is a file
copy. All shared state lives in `.product/*.md` files on the file system.

## Technical Context

**Language/Version**: Markdown (command files and templates); no runtime language required
**Primary Dependencies**: Claude Code (command host); spec-kit (for `/speckit.specify` integration)
**Storage**: File system — plain Markdown files under `.product/`
**Testing**: Manual validation of command behavior; checklist-driven spec quality review
**Target Platform**: Any OS with Claude Code installed
**Project Type**: Claude Code plugin (content toolkit — commands + templates)
**Performance Goals**: Commands complete within a single Claude response; no latency requirements
**Constraints**: Zero runtime dependencies; plain Markdown only; install by file copy
**Scale/Scope**: 4 commands, 4 templates, 4 generated documents per project

## Phase 0: Research Summary

See [research.md](research.md) for full details. Key decisions:

1. **No runtime code** — commands are Markdown files executed by Claude Code; no Python or bash needed at command time
2. **File copy installation** — zero-dependency; copies `.claude/commands/pm-*.md` and `.product/templates/`
3. **spec-kit command pattern** — YAML frontmatter (`description` + `handoffs`), `$ARGUMENTS` placeholder, Markdown body
4. **`.product/` as shared state** — each command reads what it needs, writes what it produces; no coordination protocol
5. **`[PLACEHOLDER_TOKEN]` template format** — consistent with spec-kit; HTML comments for removable instructions
6. **Prepend for release docs, merge for requirements** — `press.md` and `faq.md` prepend new releases; `requirements.md` merges by functional area

## Phase 1: Design Summary

See [data-model.md](data-model.md) and [contracts/command-contracts.md](contracts/command-contracts.md) for full details.

### Entities

| Entity | File | Owner |
|--------|------|-------|
| Domain Document | `.product/domain.md` | `/pm-domain` |
| Press Release (section) | `.product/press.md` | `/pm-press` |
| FAQ Section | `.product/faq.md` | `/pm-faq` |
| Requirements Document | `.product/requirements.md` | `/pm-align` |
| Alignment Report | (stdout only) | `/pm-align` |

### Command Interface Summary

| Command | Reads | Writes | Key Invariant |
|---------|-------|--------|---------------|
| `/pm-domain` | `domain.md` (optional) | `domain.md` | Merge-update only |
| `/pm-press` | `domain.md`, `press.md` | `press.md` | Prepend; historical releases immutable |
| `/pm-faq` | `press.md`, `faq.md` | `faq.md` | Prepend; must challenge all press claims |
| `/pm-align` | All `.product/*.md`, all `spec.md` | `press.md`, `faq.md`, `requirements.md` | Only updates upcoming sections; requirements grows monotonically |

### Constitution Check (Post-Design)

Re-evaluated after Phase 1 — all principles still satisfied:
- Principle II (Cumulative): Prepend strategy for `press.md`/`faq.md`, merge strategy for `requirements.md` both honor the invariant
- Principle III (Spec Alignment): `/pm-align` contract explicitly reads all `.specify/*/spec.md` files
- Principle V (Simplicity): Zero dependencies confirmed — no scripts, no runtimes, no services
